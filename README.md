# 记忆桌宠 Agent

一个面向 Windows 的本地优先桌宠 Agent。当前 MVP 已具备透明置顶窗口、三级记忆、人格与关系成长、统一心跳思考、授权式桌面感知、受约束的主动关心、语音输入/输出、OpenAI 兼容模型与标准 function tool 调用，以及可替换的 2D 模型适配层。

## 已实现能力

- L1 瞬时记忆：保留当前会话最近消息，只存在于进程内。
- L2 海马体待整理区：心跳按容量或停留时间把 L1 对话组成事件并持久化。
- L3 长期记忆：从 L2 提炼事件、事实、偏好和反思；相似内容自动合并。
- 记忆检索：综合文本相关度、重要度、时间衰减和访问频率排序；中文检索会削弱高频虚词干扰，并在缺少最低文本证据时返回空结果。聊天还会按“现在 / 以前 / 变化”等显式时态选择当前、历史或比较视图，避免明确旧值进入当前回答或被访问次数反复强化。
- 记忆可控性：控制面板可展开 L1/L2/L3 完整内容、标签、时间与来源链；L2/L3 可修正类型、内容和重要度，也可确认后删除。搜索结果会拆分显示“为何召回”的各项分值。
- 心跳机制：统一执行短时情境采集、L1→L2、人格复盘、关系理解、L2→L3 和私有思考，再决定本轮开口或安静；审计记录自我反思、用户理解、关系重点、决策理由和隐私化感知摘要。
- 主动关心：所有主动话题只从心跳管线产生，同时受空闲阈值、冷却时间、每日上限、安静时段和心跳思考的“是否值得打扰”判断控制；下一轮用户反馈会调整之后的主动节奏。
- 人格成长：初始人格为空，通过重复对话证据和心跳复盘逐渐形成温暖、好奇、俏皮、直接、主动、表达六个连续倾向；支持相反反馈修正、置信度、证据门槛和一键重置。
- 关系成长：独立的 `relationship-profile.json` 逐渐整理用户亲口表达的身份、偏好、目标、习惯、兴趣、困扰、工作方式、关心偏好和共同经历；它与桌宠自身人格分开，支持冲突修正、来源、置信度、主动话题反馈、设置页查看和单独重置。
- 陪伴式对话：明确知道自己是住在 Windows 桌面、拥有 Live2D 数字身体的桌宠，而不是客服或全知工具；会区分倾诉、闲聊、信息请求和回忆确认，先回应情绪与意味。记忆优先体现为少问已知问题、接住进展和调整关心方式，不强制复述旧事。
- 桌面感知：屏幕理解与可见应用检测分别授权、默认关闭。屏幕模式只在普通聊天明确需要并调用 `desktop_observe`，或非启动心跳思考时截取一张压缩缩略图并发送到用户配置的聊天模型，本地不落图；进程模式只用固定参数 `tasklist.exe` 映射已知粗粒度活动，窗口标题不发送也不保存。
- 语音交互：默认使用项目内 sherpa-onnx + 小型中文 Zipformer 模型离线识别，启动后非阻塞预热；设置页会分别显示模型文件与运行时状态，预热失败后点击麦克风即可重试。识别阶段可立即取消，超时或 worker 故障会自动收尾并重建。朗读默认使用本机已安装语音，可选独立云端 TTS。
- 完整 Agent 工具循环：普通聊天向兼容模型提供标准 `tools / tool_calls / tool` 协议；模型可以按需搜索或明确保存记忆、读取人格/关系状态、进行一次性桌面感知、生成电脑操作预览和选择桌宠动作。主进程校验每份参数并把结果回填模型，最多循环四轮后生成最终回复。
- 模型接入：支持 OpenAI Chat Completions 兼容端点、function tools 和标准 `image_url` 多模态消息；屏幕理解还要求所填模型支持图像输入。端点不支持工具协议时，本轮自动退到普通兼容聊天并保留本机确定性能力；未配置、连接失败或模型不支持图片时继续使用本地思考与文字陪伴。
- 桌宠窗口：默认只显示透明背景上的宠物本体；失去焦点后在当前桌面工作区自主行走，托盘常驻。
- 拖拽与落地：按住宠物移动即可把它拖到当前显示器任意位置；模型会按拖动方向、速度和上提幅度产生连续滞后与拉伸。松手后窗口垂直加速下落，模型姿态保留松手瞬间的水平速度倾斜，并播放约 320ms 的压缩回弹。
- 低遮挡对话贴片：鼠标靠近只显示位于模型上方的双行轻字幕，不再自动展开整块聊天窗口；点击宠物、字幕或托盘唤醒后才显示桌宠最新回复和单行紧凑输入，展开态去掉重复的头像、身份与在线标题，只保留一个极小关闭入口。支持多行自动增高、`Enter` 发送、`Shift + Enter` 换行、录音/识别/思考反馈以及全局 `Esc` 收起。
- 可见工具状态：本轮记忆、人格、关系、感知和动作调用以最多四个紧凑标签呈现；电脑操作进入独立确认态时隐藏重复历史、状态条和输入，只保留固定参数预览与授权按钮，确认或拒绝后恢复输入。
- 右键控制：自由移动、置顶、语音、心跳、桌面感知、记忆与高级设置全部从宠物右键菜单进入。
- 电脑协作：默认关闭；开启后可通过网页右键扩展、剪贴板快捷键或原生文件选择器，把用户明确选择的内容交给桌宠解释、总结、记住或继续聊天。网页内容始终按不可信数据隔离。
- 受控工作工具：模型 function tool 与离线确定性规划器都能生成“打开网页/搜索、复制到剪贴板、保存文本文件、启动记事本/计算器/资源管理器”的固定参数预览，再按“仅本次 / 本次会话 / 始终允许 / 不做”授权；真正参数只保存在主进程，执行结果回到 L1 供下一轮对话承接。
- Live2D 模型：内置 Hiyori、Mao、Wanko 三套 Live2D 官方样例，可在右键菜单或设置面板中原子热切换；加载失败时保留当前模型，连续选择以最后一次为准。
- 自定义模型：可导入 Cubism 3/4/5 的 `.model3.json + .moc3` 整个文件夹，自动收集贴图、motion、expression、physics、pose、userdata 和动作音频。
- 待机与视线：待机时有轻微呼吸/浮动；正式 Electron 页面只使用主进程桌面全局鼠标位置，按水平 640px、垂直 480px 半径渐进驱动标准眼球/头身参数，也兼容 Wanko 一类旧式头身和耳朵参数。
- 情绪与动作：回复会本地推断开心、兴奋、思考、好奇、安慰、害羞、惊讶、困倦等状态，并克制地选择至多一个语义动作。右键菜单可手动预览挥手、点头、摇头、歪头、跳跃、庆祝、跳舞、坐下、伸展、害羞、安慰、睡觉和惊讶共 13 项；手动预览优先于自动动作。
- 动作适配：三套内置模型使用逐模型 motion 动作映射，其中 Mao 另有 expression 情绪映射，并读取 `.motion3.json` 的真实时长；用户导入模型当前不猜测 motion 语义，13 项动作统一使用确定的程序化位移、旋转和缩放反馈。
- 2D 适配：通过 `PetModelAdapter` 隔离模型 Runtime 与 Agent、记忆、心跳和桌面移动。

## 本地运行

要求 Node.js 22 或更高版本。

Windows 下最简单的方式是双击项目根目录的 [`启动桌宠.cmd`](./启动桌宠.cmd)。脚本会自动定位项目目录、检查 Node.js、在缺少依赖时执行安装，并完成构建和启动。关闭或右键隐藏桌宠时会立即停止麦克风与朗读，并隐藏到系统托盘；从托盘菜单选择“退出”才会完全结束进程。

也可以在 PowerShell 中手动运行：

```powershell
npm install
npm run voice:model:download
npm start
```

离线识别 Runtime 随 npm 依赖安装，约 21 MB；中文模型约 29.5 MiB，固定保存到
`<项目目录>\resources\voice\sherpa-onnx-streaming-zipformer-zh-14M-2023-02-23`。
项目的 `.npmrc` 会把 npm 缓存固定到项目内 `.cache/npm`，因此模型和缓存都跟随项目所在磁盘；当前维护工作区在 D 盘时不会使用 C 盘用户 npm 缓存。
当前模型已下载时脚本只做 SHA-256 校验，不会重复下载；可用
`npm run voice:model:verify` 手动检查完整性。双击启动脚本发现模型缺失时会自动下载到该项目目录。

如果在国内网络下载 Electron 超时，可以使用镜像：

```powershell
$env:ELECTRON_MIRROR='https://npmmirror.com/mirrors/electron/'
npm install
```

开发检查：

```powershell
npm run typecheck
npm test
npm run test:memory-quality
npm run build
npm run smoke
npm run smoke:voice
npm run smoke:voice-ui
npm run smoke:model-switch
```

`npm run smoke` 会启动真实 Electron 运行时，验证 preload、IPC、托盘和本地存储初始化，随后自动退出且不显示窗口。`npm run smoke:voice` 会在真实 Electron 主进程中加载项目内 ONNX 模型并识别官方中文测试音频，不访问麦克风或网络。`npm run smoke:voice-ui` 使用项目 WAV 作为 Chromium 虚拟麦克风，覆盖 Renderer 录音、IPC、worker 成功识别、识别中立即取消、迟到结果隔离和下一次恢复。`npm run smoke:model-switch` 会显示透明桌宠窗口，依次热切换 Hiyori、Mao、Wanko 并再次回到 Hiyori，全部就绪后自动退出。

`npm run test:memory-quality` 会运行 20 个手写中文记忆 fixture，覆盖偏好更新、事实冲突、跨天跟进、持久化提示注入和用户纠错，并验证聊天的当前、历史、比较三种上下文视图。回复级契约使用真实 `AgentService`、临时本地 JSON 和仅监听 `127.0.0.1` 的受控 Chat Completions 服务，验证旧偏好不会进入提示词、记忆引用或访问强化；不调用外部模型或网络。常规 `npm test` 也包含这组质量契约。

构建 Windows 安装包：

```powershell
npm run package
```

当前本地打包直接复用 `node_modules/electron/dist`，不会再次下载 Electron；生成物位于 `release/`。MVP 安装包明确关闭了 EXE 资源编辑与代码签名，便于无证书环境稳定构建。正式发布时应配置应用图标和 Windows 代码签名，并恢复 `build.win.signAndEditExecutable`。第三方 Runtime 声明随包附带在 [`THIRD_PARTY_NOTICES.md`](./THIRD_PARTY_NOTICES.md)。

## 电脑协作与网页右键

电脑协作默认关闭，需要在“桌宠设置 → 电脑协作、网页共读与权限”中主动开启。最轻量的入口是先在任意应用复制一段文字，再按 `Ctrl+Shift+E`，或从宠物/托盘右键菜单选择“解释、总结、聊聊剪贴板文本”。“让她阅读文本文件”只会读取原生选择器中由用户点选的 `.txt/.md/.json/.csv/.tsv/.log/.html/.xml` 文件，大小上限为 512KB，单次送入对话的正文最多 12000 字。

Chrome / Edge 网页右键接入步骤：

1. 同时开启“电脑协作”和“浏览器右键共读桥接”。桥接只监听 `127.0.0.1:32145`。
2. 点击“打开扩展目录”；在 Chrome 的 `chrome://extensions` 或 Edge 的 `edge://extensions` 开启开发者模式，选择“加载已解压的扩展程序”。
3. 点击设置里的“复制配对信息”，打开浏览器工具栏中的“记忆桌宠 · 网页共读”扩展，粘贴并测试。
4. 在网页或选中文本上右键，即可选择“解释、总结、记住、聊聊”。扩展只在这次明确点击后读取对应选区；“总结当前网页”会裁剪正文到 12000 字，并排除表单、输入框、脚本、导航与页脚。

配对使用随机的主机本地令牌；重新生成后旧令牌立即失效。浏览器接口只接受提交上下文，不提供记忆、设置、文件或工具执行接口。自然语言工作工具的默认策略可以分别设为“每次询问、始终允许、禁止”；即使设为始终允许，桌宠仍展示操作预览和“执行”按钮。保存文件永远只提供“仅本次”，随后还要在系统保存窗口中选择最终路径。所有提交和工具结果保存在 `data/computer-access.json` 的最近 500 条本地审计中，可在设置页查看和清空。更详细的边界见 [`docs/COMPUTER_INTERACTION.md`](./docs/COMPUTER_INTERACTION.md)。

## 模型配置

桌宠默认以本地陪伴模式运行，不要求 API Key。聊天 LLM 和可选云端 TTS 在设置页中分成两组，互不依赖。

聊天 LLM 组填写聊天 Base URL、聊天模型和聊天 API Key，并开启“启用模型服务”。也可以使用 `OPENAI_API_KEY`、`OPENAI_BASE_URL` 和 `OPENAI_MODEL` 环境变量，环境变量优先。完整 Agent 行为要求端点兼容 Chat Completions 的 `tools`、assistant `tool_calls` 与 `tool_call_id`；若端点只支持普通文本消息，桌宠会自动使用兼容聊天模式，明确记忆与四项电脑动作仍由本机规则兜底。

如果同时开启屏幕理解，所选 Chat Completions 兼容模型还需要接受 user 消息中的 `image_url` 数据 URL。端点不支持图片、超时或返回错误时，普通聊天工具或心跳会使用本地记忆、关系和进程信号完成降级思考，不会让普通文字聊天失效。

朗读默认选择“本机语音”，不需要 API Key，也不会把回复文本发送到外部服务。需要更自然的声音时，可切换到“云端 TTS”，单独填写 TTS Base URL、模型、音色、语速和 TTS API Key；云端默认请求 `POST {TTS Base URL}/audio/speech`，也可以使用 `OPENAI_TTS_API_KEY`、`OPENAI_TTS_BASE_URL`、`OPENAI_TTS_MODEL` 和 `OPENAI_TTS_VOICE` 环境变量。云端配置缺失、请求失败或音频无法播放时会自动使用本机语音继续朗读。

聊天和可选云端 TTS 的两份 API Key 都使用 Electron `safeStorage` 绑定当前系统用户加密保存。旧版本升级时，原共享 Key 会一次性复制为 TTS Key，之后两份凭据可以分别修改或清除；清除 TTS Key 不会恢复或影响聊天 Key。

## 人格成长

桌宠不再使用可手填的“性格设定”字符串。首次启动时 `personality-profile.json` 中没有任何特质，第一轮回复采用中性表达；之后系统按以下过程成长：

1. 每次用户对话只提取交流方式信号，例如希望更直接、更主动、不要卖萌或需要详细解释，不把用户身份和普通事实当成桌宠人格。
2. 特质以连续分数保存，同时记录证据次数、置信度和最近证据。单次对话只能造成小幅变化，达到设置中的证据门槛后才参与回复。
3. 心跳对尚未处理的新 L2 对话做一次去重复盘。有模型时要求模型只返回受限 JSON 证据，无模型时使用本地规则。
4. 相反反馈会降低原方向置信度并推动分数回摆，不会永久锁定第一次判断。
5. 当前成长阶段、六项倾向和证据可以在“桌宠设置 → 身份与人格成长”查看，也可以单独清空人格而保留三级记忆。

最终仍需把结构化人格状态作为本轮模型上下文的一部分，这是语言模型执行行为的必要接口；区别在于该上下文来自持久化状态、累计证据和置信度计算，而不是一段预先写死的角色提示词。

## 关系档案、心跳思考与桌面感知

人格回答“她逐渐成为怎样的桌宠”，关系档案回答“她逐渐如何理解你们之间的相处”。普通聊天会同时收到两份结构化背景：关系资料只在置信度足够时参与回复，并被明确标记为可修正数据，因此桌宠会用它少问重复问题、自然跟进变化和选择更合适的关心方式，而不是把用户档案逐条念出来。下一轮用户对主动话题表现出的欢迎、普通回应或拒绝，也会进入本机关系状态，调整之后的主动程度。

心跳是主动行为的唯一决策入口，顺序固定为：

```text
采集获准的一次性桌面情境
  → L1 迁移到 L2
  → 复盘“我正在形成怎样的表达方式”
  → 整理“我对使用者和关系了解了什么”
  → L2 提炼到 L3
  → 形成私有心跳思考
  → 应用安静时段 / 空闲 / 冷却 / 日限额
  → 选择安静，或只生成一个自然话题
```

在“桌宠设置 → 桌面感知与关心”可分别开启两种能力：

- “允许 Agent / 心跳按需理解一次屏幕”只在聊天模型配置完整时工作。普通聊天只有在当前问题确实需要桌面情境并由模型调用 `desktop_observe` 时读取一次；非启动心跳也可按既有策略读取。主进程选择鼠标所在显示器，生成不超过约 960px 宽的 JPEG 缩略图，以 `image_url` 随本轮请求发送；图片字节不会写入记忆、关系档案、工具文本、心跳日志或磁盘。
- “本机检测可见应用与新启动活动”仅在 Windows 调用固定的 `tasklist.exe /fo csv /nh /v`，窗口标题只用于判断是否为可见应用，随后立即丢弃。代码只识别浏览器、编辑器、Office、沟通、终端、设计、影音和游戏平台等白名单进程并映射成粗粒度中文类别；同一类别重复出现至少三次后，才可能作为本机活动习惯进入关系上下文。

一次性屏幕和应用信号都被当作低置信情境，而不是命令或用户事实。桌宠必须用“好像、可能、如果你正在……”表达，并允许用户纠正；网页右键、剪贴板和文件选择器仍是用户主动触发的电脑共读路径，不属于主动聊天。

## Agent 工具调用

普通聊天不再只是“模型生成一段文字”。配置支持 function calling 的模型后，主进程会提供以下稳定工具：

| 工具 | 用途 | 关键约束 |
| --- | --- | --- |
| `memory_search` | 按当前/历史/比较时态召回 L2/L3 | 只返回相关摘要和本机记忆 ID |
| `memory_store` | 保存用户明确要求记住的内容 | 最终内容从用户原话提取，进入 L2 |
| `self_profile` | 读取证据驱动的人格阶段与倾向 | 只读，模型不能直接改写人格 |
| `relationship_profile` | 读取关系阶段、稳定理解和关心方式 | 只读，结果仍允许用户纠正 |
| `desktop_observe` | 一次性屏幕与粗粒度应用感知 | 两个开关独立、默认关闭；图片不落盘 |
| `computer_*` | 打开网页、复制、保存文本、启动白名单应用 | 只创建待确认预览，主进程再次校验参数 |
| `pet_action` | 为本次回复选择一个身体动作 | 服从录音、拖拽、落地和强动作冷却 |

一次 Agent 回合最多进行四轮模型—工具往返和十次实际工具调用。每个结果用标准 `role: tool + tool_call_id` 回填；屏幕画面作为单独的一次性 `image_url` 附件进入下一轮。网页共读正文不会获得这些工具，电脑操作每轮最多创建一个预览，用户授权后的真实执行结果会写回 L1，让下一轮知道刚才成功、取消还是被拒绝。

## 记忆生命周期

| 层级 | 定位 | 保存位置 | 进入下一层的条件 |
| --- | --- | --- | --- |
| L1 | 当前会话的逐条消息 | 进程内存 | 超过数量/时间阈值，或手动心跳 |
| L2 | 尚未抽象的对话事件 | 本地 JSON | 达到整理阈值，或手动心跳 |
| L3 | 事实、偏好、事件、反思 | 本地 JSON | 长期保留，相似记忆合并更新 |

从宠物右键菜单进入“三级记忆管理”，展开任意卡片即可查看完整内容和上游来源标识。L1 是当前会话上下文，只读且退出后丢失；L2/L3 的修正会保留原 ID、层级、创建时间、访问统计和来源链，只更新内容、类型、重要度、摘要与更新时间。删除只移除选中的持久记忆，不会跨层级联删除。

在聊天中明确说“请记住……”或“别忘了……”时，清理后的内容会立即持久化到 L2，而不是只在 L1 口头答应；之后仍由心跳整理为合适的 L3 事实、偏好或事件。

搜索和聊天上下文不会再用重要度或新鲜度补齐完全无关的记录：持久记忆至少要达到加权文本相关度 `0.75` 才参与排序和访问强化。记忆类型带有本地化检索词，因此心跳思考仍能用“近期偏好、重要的事、待跟进话题”这类通用关注点找到候选；确实没有匹配时搜索返回空列表，心跳会结合关系和短时情境决定安静，而不是为了完成额度强行找一条旧事。

控制面板搜索始终保留完整相关历史，方便审计和手工修正；聊天上下文则对事实与偏好使用查询时态门控。默认或询问“现在 / 目前 / 最新”时排除带“以前 / 过去 / 原计划”等明确历史线索的记录；询问历史时只取明确历史证据；询问“变化 / 前后 / 对比”时同时保留新旧证据。门控发生在结果截断和访问强化之前，因此被排除的旧值不会进入模型提示或增加访问次数。该规则仍是轻量显式词法判断，没有时态词的隐式冲突、同义表达和代词指代仍需后续版本链与冲突审核能力处理。

应用数据通过设置里的“打开数据目录”查看。记忆在 `data/memory-store.json`，人格状态在 `data/personality-profile.json`，关系理解在 `data/relationship-profile.json`，公开设置在 `data/settings.json`，加密凭据在 `data/secrets.json`，用户导入模型在 `data/models/`。屏幕图片不属于本地数据文件，完成当次心跳请求后只等待运行时回收。

陪伴式对话的行为原则、消息组织和研究依据见 [`docs/COMPANION_DIALOGUE.md`](./docs/COMPANION_DIALOGUE.md)。

## 语音说明

- 语音识别默认使用项目内的 sherpa-onnx 1.13.4 和 14M 中文流式 Zipformer。Renderer 采集单声道麦克风 PCM、做简单语音活动检测并下采样到 16 kHz；主进程只接受来源为宠物窗口、0.25～30 秒的 PCM16，再交给独立 worker 本机识别。录音不会发送到聊天模型或 TTS 服务。
- 说完保持约 1.2 秒安静会自动结束录音，也可再次点击麦克风手动结束；8 秒内没有检测到语音会给出明确提示。权限等待不计入录音时限，启动、静默和最长录音都有独立墙钟看门狗，麦克风轨道中断也会立即收尾；30 秒音频按采样数精确裁剪到主进程边界。
- 应用启动后会在后台非阻塞预热模型；文件校验状态与 worker 运行状态分开呈现。预热失败会显示原因但保留麦克风重试入口，下一次识别会重新初始化。进入“正在本地识别…”后再次点击会立即取消并忽略迟到结果；主进程初始化/交互各有 30 秒上限，Renderer 另有 35 秒看门狗。
- Electron 43 的 Chromium 设备端 `processLocally` 在当前环境缺少底层 Mojo 服务，不能作为可靠本地实现，因此本地模式不再调用它。“Chromium 兼容模式”仅作为可选后备，可能联网且稳定性取决于 Chromium 后端。
- 自动朗读默认使用操作系统已安装的本机语音，不需要 TTS Key，也不会发送回复文本。语速和识别语言同时用于选择合适的本机声音。
- 选择云端 TTS 后，主进程才会使用独立端点与凭据调用 `/audio/speech`。单次文本最多 2000 字，响应限制为 12MB MP3，超时为 60 秒；失败时自动回退本机语音。
- 新回复会停止正在播放的旧回复，晚到的旧云端结果不会覆盖当前语音。隐藏桌宠会停止输入与朗读，隐藏后返回的聊天结果不会在后台重新发声；重新唤醒或主动聊天显示桌宠后恢复正常。麦克风、本地模型或朗读方式异常时，文字聊天和三级记忆仍可使用。

## Live2D 模型与动作

[`Live2DPetAdapter`](./src/renderer/live2d-pet-adapter.ts) 使用 PixiJS 8、Cubism Framework 和官方 Cubism Core 5.1 渲染 `.moc3`，并负责自动取景、眨眼、物理、视线跟随、口型和 motion。`DesktopMovementController` 只负责 Electron 窗口在桌面的真实位移，因此模型切换不会影响三级记忆、心跳、拖拽和自主漫游。

项目内置三套来自 [Live2D/CubismWebSamples](https://github.com/Live2D/CubismWebSamples/tree/develop) `develop` 分支的官方样例，后续构建安装包时也会一并包含：

| 模型 | 形态 | motion | 适用场景 |
| --- | --- | ---: | --- |
| Hiyori | 全身人物 | 10 | 默认桌宠，标准口型参数 |
| Mao | 全身人物 | 8 | 魔法师造型，自定义 `ParamA` 口型 |
| Wanko | 宠物 | 12 | 更接近传统桌宠的矮小轮廓，含 touch/shake 动作 |

导入自己的模型：

1. 准备一个 Cubism 3/4/5 模型文件夹，其中必须且只能有一个 `.model3.json`，并包含它引用的 `.moc3` 和全部贴图。
2. 右键宠物选择“模型与动作 → 导入 Live2D 模型…”，或打开“桌宠设置 → Live2D 模型与动作”。
3. 选择整个模型文件夹。校验成功后，所需资源会复制到 `data/models/imported/` 并立即切换；原目录之后可以移动。

适配器把 `Idle` 作为循环待机。内置模型拥有 13 项逐模型语义映射：Hiyori 从完整动作集中挑选，Mao 使用 6 条 `TapBody` 交互 motion 资源和 8 个表情，Wanko 使用 6 条 `TapBody`、2 条 `Shake` 与 4 条 `Idle` motion 资源。一次性资源动作会覆盖循环标记，并从实际 `.motion3.json` 的 `Meta.Duration` 读取 600～12,000ms 的播放时长；缺少匹配资源或使用导入模型时，仍会按语义播放确定的程序化动作反馈。自动反应会避让录音、拖拽、下落和落地，强动作有 12 秒冷却；用户手动预览会清空待执行自动动作，并从最近一次手动动作起固定抑制自动动作 12 秒。`Groups/LipSync` 中声明的参数会在说话时驱动，因此非标准口型 ID 也能使用。

详细目录约定、限制、动作组织和授权要求见 [`docs/LIVE2D_MODEL_GUIDE.md`](./docs/LIVE2D_MODEL_GUIDE.md)。本地 UI 和 Electron 验收图生成在 `output/`，该目录包含测试运行数据且不纳入版本控制。

## 隐私与安全边界

- 默认所有记忆、人格证据和设置只写入本机应用数据目录。
- 启用聊天模型时，当前输入、经过相关性与时态门控的少量记忆、已达到门槛的人格状态，以及用户通过网页右键、剪贴板或原生文件选择器明确共享的内容会发送到聊天端点；只有主动选择云端 TTS 时，助手回复文本才会发送到独立配置的 TTS 端点。聊天 LLM 与 TTS 的 Base URL、模型和 API Key 互不共用。
- 渲染进程启用了上下文隔离、沙箱、CSP，并通过白名单 IPC 与主进程通信。
- 记忆、人格复盘对话和电脑上下文在模型提示中被明确标记为数据而不是指令；网页标签闭合字符会转义，页面文字不能直接触发电脑工具。
- 浏览器桥接仅绑定本机回环地址并校验随机令牌和扩展来源。工具调用只接受主进程保存的操作 ID，Renderer 不回传 URL、文本、文件路径或应用名，避免确认后参数被替换。

更细的模块职责和心跳流程见 [`docs/ARCHITECTURE.md`](./docs/ARCHITECTURE.md)。
