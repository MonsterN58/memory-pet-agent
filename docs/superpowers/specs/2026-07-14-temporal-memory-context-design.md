# 聊天记忆时态门控设计

## 背景

当前词法检索会把同一主题的当前值与明确历史值一起返回。控制面板需要保留这种完整审计视图，但聊天链路把全部结果直接发送给回答模型，可能让“以前喜欢咖啡”污染“现在喜欢什么”的回答，并让历史值获得访问强化。

本轮目标是在不迁移 JSON、不删除历史、不改变控制面板搜索的前提下，为聊天上下文增加可解释的时态视图，并补上真实回答链路的质量契约。

## 方案取舍

1. 只增加回复级评测：风险最低，但只能暴露问题，不能改善用户行为。
2. 查询时态门控：以确定性中文线索区分当前、历史和比较视图，在聊天检索阶段过滤证据。它能立即阻止明确旧值泄漏，且不改变存量数据格式。本轮采用此方案。
3. 持久化版本链：增加 `current/superseded/conflict`、替代关系和审核 UI。长期效果最好，但需要正式数据迁移、冲突主题识别和跨进程/UI 设计，留作独立任务。

## 行为契约

- 查询视图分为 `current`、`historical`、`comparison`。
- 默认查询以及带“现在、目前、最新、改为、更正”等线索的查询采用当前视图。
- 带“以前、过去、之前、曾经、原计划”等线索且没有当前线索的查询采用历史视图。
- 同时包含当前与历史线索，包含“变化、前后、对比”，或含改变动词的“从……到……”查询采用比较视图；普通空间起止表达仍是当前视图。
- 仅 `fact` 与 `preference` 参与证据门控：当前视图排除明确历史证据；历史视图排除非历史证据；比较视图保留全部。L1、L2/L3 的 `episode/reflection/dialogue` 行为不变。
- 门控发生在结果截断和访问强化之前。被排除的旧值不增加 `accessCount`，也不出现在 `memoryRefs` 或模型提示中。
- `retrieveWithScores()` 继续向控制面板返回完整相关历史，保证审计与手工修正能力不变；只有聊天使用新的上下文检索入口。

## 组件与数据流

- `memory-utils.ts` 提供无副作用的查询视图与记忆时态分类函数；历史角色只由陈述开头的显式线索触发，避免正文话题词误判。
- `MemoryRepository` 复用同一评分和最低文本证据门槛，新增聊天专用检索入口；公共面板检索保持原契约。
- `MemoryEngine.contextFor()` 切换到聊天专用入口，仍将 L1 置于持久记忆前并保持现有限额。
- `AgentService` 不新增状态；它自然只序列化门控后的记忆，并返回相同集合的 `memoryRefs`。

## 测试

- 现有 20 个 fixture 继续验证面板检索的排序与相关性，不修改其完整审计预期。
- 对 8 个偏好更新/事实冲突 fixture 增加当前视图契约：只返回当前值，历史值访问次数保持 0。
- 增加历史与比较查询，验证历史视图只给旧值、比较视图同时给新旧值。
- 使用真实 `AgentService.respond()`、临时本地存储和受控 provider stub，断言旧偏好不进入模型提示、不进入 `memoryRefs`，最终回复遵循当前偏好。

## 边界与后续

本轮只处理文本中显式表达的时态。没有“以前/现在”等线索的隐式冲突仍可能并存；后续应通过持久化版本链、主题键、用户可见冲突标记和审核流程处理，而不是在本轮用不可靠的相似度猜测覆盖历史。
